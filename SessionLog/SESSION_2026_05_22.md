**`ListingPage.tsx` props:**
```typescript
type Props = {
  products: MappedProduct[]       // pre-filtered product list
  title: string                   // page heading (e.g. "Heckler & Koch")
  eyebrow?: string                // small label above title (e.g. "Brand")
  breadcrumbs: { label: string; href?: string }[]
  hideBrandFilter?: boolean       // hide brand filter (used on brand pages)
  hideCategoryFilter?: boolean    // hide category filter (used on category pages)
  basePath: string                // e.g. "/brand/heckler-koch"
}
```

**Key design decisions:**
- `basePath` is critical — URL filter sync appends `?brand=...&caliber=...` to the `basePath` instead of navigating to `/shop`. Without this, every filter change on `/brand/heckler-koch` would redirect the user to `/shop?brand=...`.
- Brand pages hide the brand filter (you're already on the brand page, the filter is redundant).
- Category pages hide the category filter for the same reason.
- Collection pages hide the brand filter (collections can span multiple brands; the brand filter would be confusing).

**Filter scroll fix (applied to both `ShopPage.tsx` and `ListingPage.tsx`):**
- The outer filter section container previously had `maxHeight: "500px"`, which hard-cut the list at approximately 19 items. Smith & Wesson (alphabetically in the "S" group) was invisible — completely cut off.
- Fixed by setting the outer container to `maxHeight: "9999px"` (no effective cap) and adding a scrollable inner container: `maxHeight: "340px"`, `overflowY: "auto"`, `scrollbarWidth: "thin"`, `paddingRight: "12px"`.
- The `paddingRight: "12px"` was added because without it the thin scrollbar overlapped the item count numbers (e.g. "Korth (18)") making them unreadable.

**`MappedProduct` type extended in `src/lib/medusa.ts`:**
- Added `collection_id: string | null`
- Added `collection_handle: string | null`
- These are populated from `p.collection?.id` and `p.collection?.handle` in `mapMedusaProduct()`

### Smith & Wesson Slug Fix
The `toSlug()` function had a bug where "Smith & Wesson" produced the slug `"smith-wesson"` but if the brand name was stored as "Smith and Wesson" it would produce `"smith-and-wesson"` — a different slug. The brand page at `/brand/smith-wesson` existed but "Smith and Wesson" would fail to match it.

**Fix:** Added `&` ↔ `and` equivalence to `toSlug()`:
```javascript
function toSlug(str) {
  return str
    .toLowerCase()
    .replace(/&amp;/g, 'and')      // decode HTML entity first
    .replace(/\s*&\s*/g, '-')      // "Smith & Wesson" → "smith-wesson"
    .replace(/\s+and\s+/g, '-')    // "Smith and Wesson" → "smith-wesson"
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/-+/g, '-')
    .replace(/^-|-$/g, '')
}
```

## PART 7 — CACHE REVALIDATION
**Commit: `a00c7e8`**

### What Was Built

**`src/app/api/medusa-hook/route.ts`** — Medusa webhook endpoint

When a product is created, updated, or deleted in the Medusa admin, Medusa automatically sends a POST request to this endpoint. The endpoint then tells Next.js to clear its product cache so the storefront updates within seconds instead of waiting for the 60-second ISR window.

**Handles these Medusa events:**
- `product.created`, `product.updated`, `product.deleted`
- `product-variant.created`, `product-variant.updated`, `product-variant.deleted`
- `product-category.created`, `product-category.updated`, `product-category.deleted`
- `product-collection.created`, `product-collection.updated`, `product-collection.deleted`

**Security:** Optionally protected by `REVALIDATE_SECRET` environment variable. The secret can be passed as `?secret=VALUE` in the webhook URL or as an `x-revalidate-secret` header.

**Setup in Medusa admin:** Settings → Webhooks → Add webhook:
- URL: `https://dev.luxus-collection.com/api/medusa-hook?secret=YOUR_SECRET`
- Events: select all product-related events

**Important Next.js 16 note:** `revalidateTag()` in Next.js 16 requires a second argument:
```typescript
revalidateTag("products", { expire: 0 })  // correct for Next.js 16
revalidateTag("products")                  // wrong — throws error in Next.js 16
```

## PART 8 — MULTI-BRAND PRODUCT FIX (ATTRIBUTES MODULE DISCOVERY)
**Commits: `f7e912d`, `df67176`, `fb0d08d`, `10e4448`, `6f27dee`, `4a35970`, `8f4d80c`, `5b88aa5`, `db60eae`**

### The Problem
"10-8 Performance" was not appearing in the brand filter on the shop page. The user confirmed both "Hilton Yam" and "10-8 Performance" had been selected in the Medusa admin for the product "Hilton Yam Custom Delta 1911" for a long time.

### Investigation

**Step 1 — Debug endpoint built (`f7e912d`, `df67176`)**

Created `/api/debug-products` — a diagnostic endpoint that bypasses all Next.js caching and fetches raw data directly from Medusa. Confirmed that `metadata.brand` for the Hilton Yam product only contained `"Hilton Yam"` — a plain string with only one value. "10-8 Performance" was completely absent from metadata.

**Step 2 — Ruled out frontend parsing issues (`fb0d08d`, `10e4448`, `6f27dee`)**

Initially suspected that the frontend `readAttr()` function was failing to parse multi-value brand data. Updated `readAttr()` to handle:
- Real arrays: `["Hilton Yam", "10-8 Performance"]`
- JSON-stringified arrays: `'["Hilton Yam","10-8 Performance"]'`
- Comma-separated strings: `"Hilton Yam, 10-8 Performance"`

This did not fix the issue because "10-8 Performance" was never in the metadata at all — the parsing improvements were irrelevant for this case (though useful for other products).

**Step 3 — Checked custom attributes module (`4a35970`, `8f4d80c`, `5b88aa5`)**

Updated the debug endpoint to:
1. Fetch the custom attributes endpoint for the Hilton Yam product: `GET /store/products/:id/attributes`
2. Test whether `*attribute_values,*attribute_values.attribute_type` could be included in the main products list query as a field expansion

**Debug log 3 confirmed the root cause:**

The attributes endpoint returned both brands correctly:
```json
{
  "attribute_values": [
    { "value": "10-8 Performance", "attribute_type": { "slug": "brand" } },
    { "value": "Hilton Yam",       "attribute_type": { "slug": "brand" } },
    { "value": "1911",             "attribute_type": { "slug": "model" } }
  ]
}
```

Field expansion test confirmed: `*attribute_values,*attribute_values.attribute_type` works in the main `/store/products` query. The attribute type object includes a `slug` field ("brand", "model", "caliber", "action", "barrel-length") that can be used to identify attribute types without hardcoding IDs.

### Root Cause
The Medusa admin `product-attributes.tsx` widget saves all attribute data (brand, model, caliber, action, barrel length) to the **custom attributes module**. This module supports multi-select — a product can have multiple brands, multiple calibers, etc.

The `metadata.brand` field is a legacy/secondary field. It only ever holds a single string value. When the admin saves, it writes to the attributes module; it does not reliably sync to metadata.

All frontend code was reading only from `metadata.brand`, completely ignoring the attributes module. This meant:
- Multi-brand products only showed one brand
- Any attribute set in the admin might not match what the storefront displays
- The brand filter was incomplete

### The Fix (`db60eae`)

**`src/lib/medusa.ts` — two new functions:**

```typescript
// Groups attribute_values by attribute_type.slug
function buildAttrMap(p: any): Record<string, string[]> {
  const map: Record<string, string[]> = {}
  for (const av of (p.attribute_values ?? [])) {
    const slug = av.attribute_type?.slug
    if (!slug || av.value == null) continue
    const val = String(av.value).trim()
    if (!val) continue
    if (!map[slug]) map[slug] = []
    if (!map[slug].includes(val)) map[slug].push(val)
  }
  return map
}

// Prefers attribute_values over metadata; handles hyphen/underscore slug variants
function pickAttr(attrMap, slug, metaFallback) {
  const variants = [slug, slug.replace(/-/g, '_'), slug.replace(/_/g, '-')]
  for (const s of variants) {
    if (attrMap[s]?.length) return readAttr(attrMap[s])
  }
  return readAttr(metaFallback)
}
```

**`mapMedusaProduct()` updated:**
```typescript
const attrMap      = buildAttrMap(p)
const brand        = pickAttr(attrMap, "brand",        p.metadata?.brand)
const model        = pickAttr(attrMap, "model",        p.metadata?.model)
const caliber      = pickAttr(attrMap, "caliber",      p.metadata?.caliber)
const action       = pickAttr(attrMap, "action",       p.metadata?.action)
const barrel_length = pickAttr(attrMap, "barrel-length", p.metadata?.barrel_length)
```

**All product fetch field strings updated** to include `*attribute_values,*attribute_values.attribute_type` in:
- `src/app/shop/page.tsx`
- `src/app/brand/[slug]/page.tsx`
- `src/app/category/[slug]/page.tsx`
- `src/app/collection/[slug]/page.tsx`
- `src/app/product/[handle]/page.tsx` (both FIELDS and RELATED_FIELDS)
- `src/app/page.tsx` (home page)
- `src/lib/api.ts` — `getProduct()` function

## PART 9 — DROPDOWN HOVER FIX
**Commit: `02ba357`**

### The Problem
The "Shop By" and "Contact" dropdown menus in the desktop header would close as soon as the user moved their mouse toward the dropdown panel. This made them nearly unusable.

### Root Cause
Both dropdowns were positioned with `top: "calc(100% + 14px)"` — placing the panel 14 pixels below the bottom of the nav item. The `onMouseEnter` / `onMouseLeave` event handlers were on the wrapper `div` that contained the nav item text.

When the mouse moved downward from the nav item toward the dropdown panel, it crossed the 14px gap. This gap was **outside** the wrapper `div`. As soon as the mouse entered the gap, `onMouseLeave` fired on the wrapper, setting the open state to false and hiding the dropdown before the mouse could reach it.

### The Fix
Wrapped each dropdown panel in a transparent outer `div` that starts at `top: "100%"` (immediately below the nav item) and has `paddingTop: "14px"`. The outer div is transparent and has no background — it just extends the hover zone to cover the gap.

All visual styling (background color, border, box shadow) was moved to an inner `div`.

```jsx
{/* Before — gap was outside hover zone */}
<div style={{ position: "absolute", top: "calc(100% + 14px)", background: "...", border: "..." }}>
  {items}
</div>

{/* After — transparent outer div covers the gap */}
<div style={{ position: "absolute", top: "100%", paddingTop: "14px" }}>
  <div style={{ background: "...", border: "..." }}>
    {items}
  </div>
</div>
```

The visual appearance is identical — there is still a 14px visual gap between the nav item and the dropdown panel. But now the mouse can travel through that gap without closing the menu.

Applied to both the "Shop By" dropdown and the "Contact" dropdown.
